# Perama Table 2 / CDR Method Investigation

## Summary

This note investigates how the CDR_75ms values in Table 2 of the Perama Shipyard Wi-Fi study may have been generated from the author's public repository. No official Table 2 exporter or explicitly named CDR formula was found. The repository suggests a likely steady-state control-delay ratio logic using a 75 ms threshold over approximately the [120,181] s window, with about 120 expected control packets as denominator. Therefore, the Perama baseline keeps the paper Table 2 values as the primary CDR source, while the laboratory workflow defines its own reproducible raw-log CDR formula.

Repository investigated:
`https://github.com/minarady1/wifi_for_industrial_robotics`

Local clone:
`local clone of the author repository`

Investigation scope:
read-only search and analysis of the author repository to identify how the paper Table 2 CDR_75ms values may have been calculated.

## 找到的相关文件

### `run.bash`

`run.bash` starts the measurement components. The relevant ROS2 control command is:

- `ros2 run tx_control listener ... "robot_control" ...`

This indicates that ROS/control delay data are generated by the ROS2 listener during each run.

### `ros2_ws/src/tx_control/tx_control/publisher_member_function.py`

Relevant behavior:

- Publishes control messages every `0.5` seconds.
- Payload includes sequence number and send timestamp in nanoseconds.

This implies an intended control rate of 2 Hz. For a 60 s steady window, the expected packet count is approximately 120.

### `ros2_ws/src/tx_control/tx_control/subscriber_member_function.py`

Relevant behavior:

- Subscribes to `robot_control`.
- Parses `seqnum` and source timestamp.
- Calculates receive-side delay as receive timestamp minus source timestamp.
- Logs receive timestamp and original payload.

The subscriber log is the source used by `preprocessor.py` to compute `control_delay_ms`.

### `plots/preprocessor.py`

Relevant behavior:

- `parse_control_log_line` computes:
  - `delay_ms = (receive_timestamp_ns - src_timestamp_ns) / 1e6`
  - `loss = current_seqnum - previous_seqnum - 1`
  - `timestamp_s` relative to the first received control record.
- `parse_control_log` writes:
  - `control_datetime`
  - `control_delay_ms`
  - `control_loss`
  - `control_timestamp_s`
- PHY, PTP, and iperf parsers can write `no_conn` and `no_data`.
- The control parser does not appear to write `no_conn` or `no_data`; missing control packets are represented indirectly by absent received records and `control_loss`.

### `plots/plotter.py`

Relevant behavior:

- `clean_list` removes `no_conn` and `no_data`.
- `clean` removes `no_conn` and `no_data` from paired time/data arrays.
- `trim_time` keeps samples inside a configured time range.
- `consecutive_overdelay(data_list, threshold)` counts consecutive samples where delay exceeds a threshold.
- Control delay plotting is configured with:
  - variable: `control_delay_ms`
  - time variable: `control_timestamp_s`
  - x-axis limit: `[120,181]`
  - trim range: `[120,181]`
- In `plot_cdf`, the code calculates:
  - `pdr = len(sanitized_data) / 120`, capped at 1.
  - a ratio based on samples with delay below `75`.
  - a commented print for the product of these two terms.

This is the closest code-level evidence for the Table 2 CDR_75ms calculation.

## 是否找到 Table 2 生成脚本

No direct Table 2 generation script was found.

There is no script found that explicitly writes the paper's Table 2 values, names Table 2, or exports a CDR table.

The closest candidate is `plots/plotter.py`, especially the control-delay CDF logic and the commented product of:

- received-sample ratio in the steady window, and
- fraction of received samples below 75 ms.

## 是否找到 CDR 公式

No formula explicitly named `CDR` was found.

However, the closest recoverable formula from `plots/plotter.py` is:

```text
pdr = min(received_control_samples_in_steady_window / 120, 1)
under_75_ratio = count(delay_ms < 75) / received_control_samples_in_steady_window
candidate_CDR_75ms = pdr * under_75_ratio * 100
```

When the number of received samples in the steady window is at most 120, this simplifies to:

```text
candidate_CDR_75ms = count(delay_ms < 75) / 120 * 100
```

This candidate matches the paper Table 2 values very closely:

- 36 / 36 rows are within 2 percentage points.
- Maximum absolute difference is about 0.5 percentage points.
- 32 / 36 rows match after rounding to the nearest integer.

This strongly suggests that Table 2 CDR_75ms used the steady-window expected-packet denominator of 120, not only the received-packet denominator.

## 75ms threshold 是否在代码中出现

Yes.

The 75 ms threshold appears in `plots/plotter.py` in two relevant places:

- `consecutive_overdelay(..., 75)` for over-delay burst analysis.
- CDF-related logic that counts samples with delay below `75`.

The code uses a strict below-threshold comparison for the CDF-related ratio:

```text
delay_ms < 75
```

## steady state 口径是否能确认

Mostly yes.

The control delay plotting configuration in `plots/plotter.py` uses:

```text
trim_time = [120,181]
```

and the corresponding x-axis limit is also:

```text
xlimit = [120,181]
```

PTP path delay plotting uses the same steady-state window. This supports the interpretation that the author's steady-state analysis window is approximately:

```text
120 s <= time_s <= 181 s
```

In our own reporting, this can be stated as "time_s >= 120 s steady-state window, with the author's plot code using [120,181] for Table-2-like control-delay analysis."

## 分母是否能确认

It can be inferred strongly, but it is not documented as a named Table 2 denominator.

Evidence:

- The ROS2 publisher sends one control message every 0.5 s.
- A 60 s steady window gives about 120 expected packets.
- `plots/plotter.py` computes `pdr = len(sanitized_data) / 120`, capped at 1.
- The CDF logic combines this PDR-like term with the below-75-ms received-sample ratio.
- This candidate formula closely reproduces Table 2.

Therefore, the most likely denominator is:

```text
expected control packets in the steady window, approximately 120
```

Not simply:

- all sent packets over the whole run,
- all received packets over the whole run,
- or only received packets in the steady window.

## no_data/no_conn 处理是否能确认

Partly yes.

In preprocessing:

- PHY, PTP, and iperf parsers can insert `no_conn` and `no_data`.
- Control parser does not appear to insert `no_conn` or `no_data`.

In plotting:

- `clean_list` and `clean` remove `no_conn` and `no_data` values before numeric calculations.

For control delay specifically:

- Missing control messages are not represented as `no_data` or `no_conn` in `control_delay_ms`.
- They are represented by fewer received control samples and by the `control_loss` field derived from sequence gaps.

## negative delay 处理是否能确认

The code does not show explicit negative-delay filtering for CDR-like logic.

Negative numeric delay values remain numeric after cleaning because `clean` removes only `no_conn` and `no_data`. In the below-75-ms check, negative values would count as successful because they satisfy:

```text
delay_ms < 75
```

This is important for our later use: negative ROS/control delay should be diagnosed separately and should limit the confidence of ROS/control delay as a main KPI.

## 丢包是否算 CDR 失败

Most likely yes, through the expected-packet denominator.

The formula inferred from `plots/plotter.py` effectively penalizes missing received control packets because:

```text
pdr = received_samples / 120
```

If fewer control messages are received in the steady window, the CDR_75ms value decreases even if all received messages are below 75 ms.

The `control_loss` field is calculated from sequence-number gaps, but the CDR-like formula does not appear to directly sum `control_loss`. The penalty comes from the reduced number of received samples in the steady window.

## 为什么我们之前 JSON 反算和 Table 2 只部分匹配

The most likely reason is a denominator and window mismatch.

If CDR is calculated as:

```text
count(delay_ms < 75) / received_control_samples
```

then packet loss or missing received samples are not penalized. This creates mismatches, especially for cases like `S3 / ac/5/20`, where the paper Table 2 reports 0.

If CDR is calculated over all time instead of the author's steady window `[120,181]`, it also mismatches Table 2.

The closest reproduction uses:

```text
steady window: 120 s <= control_timestamp_s <= 181 s
threshold: delay_ms < 75
expected denominator: 120 control packets
```

or equivalently the product:

```text
min(received_samples / 120, 1) * count(delay_ms < 75) / received_samples
```

For `S3 / ac/5/20`, there are no received control samples in the steady window, so the reproduced value is 0, matching Table 2.

For `S3 / ax/6/20`, the steady-window count is about 61 received samples and about 59 below 75 ms, giving:

```text
59 / 120 * 100 = 49.17%
```

which matches the Table 2 value of 49 after rounding.

## 对我们自己实验室 Wi-Fi5 实测的建议 CDR 公式

For our own laboratory Wi-Fi5 measurements, use an explicit and auditable formula instead of relying on inferred plotting behavior.

Recommended primary formula:

```text
CDR_75ms_percent =
100 * successful_control_packets_in_steady_window / expected_control_packets_in_steady_window
```

where:

```text
successful_control_packets_in_steady_window =
count(received packet with valid timestamps and 0 <= delay_ms < 75)
```

and:

```text
expected_control_packets_in_steady_window =
configured_control_rate_hz * steady_window_duration_s
```

For direct comparability with the author's inferred method, also report:

```text
author_style_CDR_75ms_percent =
100 * count(received packet with delay_ms < 75) / 120
```

for a 2 Hz, 60 s steady window.

Important recommendation:

- Keep negative delays as a separate diagnostic category.
- Do not silently count negative delays as successful in our main laboratory CDR.
- Report both strict laboratory CDR and author-style reproduced CDR if comparison to Table 2 is required.

## Bottom line

The author repository does not contain a direct Table 2 exporter or an explicitly named CDR formula.

The strongest code-supported reconstruction is:

```text
CDR_75ms = 100 * min(received_control_samples_120_to_181 / 120, 1)
                 * count(delay_ms < 75) / received_control_samples_120_to_181
```

with a zero result when no steady-window control samples exist.

For the normal case where received steady-window samples are at most 120, this is effectively:

```text
CDR_75ms = 100 * count(delay_ms < 75 in [120,181]) / 120
```

This explains why JSON reverse calculations based only on received packets, or based on all-time samples, match Table 2 only partially.
